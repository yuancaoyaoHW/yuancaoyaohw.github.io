01
Executive Summary

执行摘要

先给结论，再展开证据与工程细节。

研究结论

Megakernel 的本质

它不是“更大的算子融合”那么简单，而是把大量模型任务放入一个长期驻留的 GPU 内核中，通过设备侧解释器、任务队列、显式事件/计数器与 warp 专职化来调度。目标是减少 CPU 发射空隙、跨内核全局屏障、尾波浪费与中间张量往返 HBM，并把计算、访存和通信细粒度重叠。

官方规格

Rubin 的本质

Rubin 不只是一颗 GPU，而是一套机架级系统：Rubin GPU 提供 288 GB HBM4、22 TB/s HBM 带宽和 3.6 TB/s NVLink 6；Vera CPU 提供大容量一致性内存；NVL72 把 72 颗 GPU 组织成约 260 TB/s 机架内互联域。[R1][R2][R3]

架构推断

二者为何可能相互放大

Rubin 大幅压缩计算、HBM 与链路时间后，固定的发射、同步和调度气泡更容易成为总时延中的显著比例；Megakernel 正好针对这些“非算术成本”。同时，Rubin 的 TMA 内联描述符更新、设备发起 NVLink counted writes 与更细粒度的依赖触发机制，为动态 MoE 和跨 GPU 设备侧运行时提供更合适的硬件原语。[R1]

证据边界

不能直接声称多少倍

公开 Megakernel 数据来自 A100、H100、H200、B200 或 L20；Rubin 数据主要是官方峰值、厂商模型与少量合作伙伴现场测量。二者尚无公开、可复现实验交集，因此本报告只给出机制级判断、性能下界和验证方案，不伪造联合加速倍数。

一句话判断： 对小批量、内存受限、算子碎片多、跨 GPU 通信可细粒度重叠的推理，Megakernel 与 Rubin 的组合最值得投入；对大批量、单个 GEMM 已长期占满张量核心的工作，收益更可能来自 Rubin 本身，Megakernel 应采用混合而不是“全包”策略。

Scope & Evidence

范围、定义与证据分级

本报告中的“Megakernel”

这里专指 GPU/大模型系统中的 模型级或大图级持久内核执行模式：每张 GPU 启动一个或少数长期驻留的内核，在内核内部完成任务解释、依赖管理、数据搬运、计算和通信。它与数据库、图形渲染或操作系统中偶尔出现的同名概念无关。

实测

论文、公开代码或合作伙伴在真实硬件上给出的测量。仍需核对模型、批量、精度、提示分布、服务框架与统计口径。

论文/原型

研究原型的算法与性能，通常可解释性强，但工程成熟度、可移植性和服务功能可能不及生产框架。

厂商峰值/预测

芯片规格、峰值算力、路线图或厂商模型。用于理解上限与方向，不等于应用端持续性能。

架构推断

由公开机制推导的协同关系。必须通过 Rubin 实机、相同模型和相同服务约束验证。

对比规则： 本报告不会把“batch 1 延迟”“离线吞吐”“tokens/MW”“峰值 PFLOPS”画在同一条排名轴上。每个数字都保留其基准条件；跨论文数字只能说明趋势，不能当成统一排行榜。
03
Megakernel Principles

Megakernel：从算子流水线到设备侧运行时

把“执行模型”从 CPU 主导的内核序列，迁移成 GPU 内部的任务系统。

3.1 为什么传统执行会出现“气泡”

典型深度学习框架把归一化、矩阵乘、激活、注意力、KV 读写与集合通信分别映射为 CUDA 内核。内核边界提供了方便的全局同步语义，但也引入 CPU 发射延迟、队列空隙、每个算子的尾波、重复读取/写回 HBM，以及无法跨边界调度的空闲 SM。小批量自回归解码中，单个算子本来就很短，几微秒级间隙可累积成可见时延。Hazy Research 对 Llama-1B 的分析估算每层约 7 次发射、16 层、约 5 μs 间隙时，仅发射停顿就会形成约 560 μs 量级的成本。[M1]

概念图：真实实现会按 tile、SM、warp 和通信阶段进一步拆分。

3.2 运行时结构：宿主编排 + 设备解释/调度

① 图拆分把模型算子切成 GEMM tile、归一化片段、attention block、load/store、collective 等任务。
② 编译/排程主机端生成每 SM 指令序列或全局任务 DAG，决定分块、依赖和资源预算。
③ 一次发射每 GPU 启动持久内核；网格长期驻留，循环读取任务。
④ 设备执行解释器或去中心化调度器分配 warp；显式事件表示 producer/consumer 关系。
⑤ 细粒度重叠让加载、张量核心计算、写回和 NVLink 通信跨操作并发。

Hazy 的低延迟实现采用主机预生成的 per-SM 指令序列与 GPU 解释器；高吞吐实现则进一步引入全局工作队列，并把 loader、consumer、storer、controller 线程组专职化。MPK 将类似思想封装为 torch.compile(backend=MPK) 的编译后端；Event Tensor/ETC 则用事件张量抽象描述动态依赖和数据相关形状。[M1][M2][M3][M4]

典型逻辑结构。不同实现对静态/动态调度、内存分配和通信原语的选择不同。

3.3 调度、同步与内存层次

静态 per-SM 指令流

编译期把工作精确分配给每个 SM，运行时开销低、行为可预测，适合形状稳定和 batch 很小的模型。代价是对动态序列长度、MoE 路由和负载偏斜不够鲁棒。

动态全局工作队列

SM 完成一块任务后领取下一块，可缓解尾波与不均衡，适合大 batch、异构任务和 MoE；但原子队列、争用、优先级与公平性会成为新成本。

混合层级调度

外层按 phase/专家/请求动态分配，内层按 tile 静态流水。通常更接近生产需求：既保留热点路径效率，又处理动态形状和连续批处理。

共享内存页化

把有限 SMEM 切成可复用页，生产者和消费者通过页所有权/计数器传递中间结果。Hazy 低延迟原型曾将约 213 KB 资源组织为 13 个约 16 KiB 页面。[M1]

事件/计数器依赖

用细粒度发布与等待替代内核边界。正确性依赖内存顺序、栅栏范围、环形缓冲复用和异常路径；错误可能表现为罕见数据竞争或死锁。

软件流水与跨操作重叠

下一 tile 的载入可与当前 tile 的 Tensor Core 运算并行；通信 warp 可在未来计算推进时发送前一结果。高吞吐原型的消融显示，仅 pipelining 和 interleaving 就可贡献数个百分点。[M2]

3.4 它与算子融合、CUDA Graph、Persistent Kernel 有何不同

技术控制面内核边界主要收益最适合主要局限
算子融合编译器/模板融合区之外仍存在减少中间读写与发射局部、规则的数据流资源压力与组合爆炸；难跨大算子
CUDA GraphCPU 录制/重放仍是多个内核显著降低 CPU 发射开销形状稳定的重复图内核间全局边界和 HBM 往返仍在；动态性受限
Persistent Kernel设备循环一个或少数内核避免重复启动、复用状态队列、扫描、通信、单类算子不必然具备模型级任务语义
Megakernel设备侧解释器/调度器覆盖大图或整步执行跨算子流水、细粒度依赖、设备通信碎片化推理、动态任务图、TP/MoE编译与验证复杂；资源、QoS、可移植性挑战
工程上常是组合而非替代： Megakernel 内部仍会调用/内联高度优化的 GEMM/attention tile；外部可能继续使用 CUDA Graph 管理预处理、采样或无法纳入的库调用。最优系统通常是“设备侧大图 + 少量专用内核”的混合结构。

4.2 什么时候会变快，什么时候不会

高概率受益

batch 1–小 batch；每 token 数十到上百个短内核；低 HBM 利用；大量 elementwise/norm/KV 搬运；SM 尾波明显；通信可分块并与后续计算重叠；请求形状足够稳定或编译器可表达动态性。

收益有限或可能退化

大 batch 下单个 GEMM 已长时间满载；库内核本身已达高效率；Megakernel 导致寄存器/SMEM 过量而降低 occupancy；动态队列争用；连续批处理/采样仍依赖 CPU；多租户要求抢占和隔离；collective 已不可隐藏。

专家注：为什么“单个大内核”不等于“高性能”
内核做得更大，会同时放大寄存器、共享内存、代码体积和指令缓存压力。若每 CTA 占用资源过多，驻留 CTA 数下降，反而失去延迟隐藏；若同步图设计不当，warp 会在事件上空转。Megakernel 的难点是资源预算与调度，而不是把函数机械地拼接到一起。

NVIDIA Rubin Architecture

Rubin：从双芯粒 GPU 到 NVL72 机架

理解 Rubin 必须同时看 GPU、CPU、一致性互联、机架内 NVLink 与 scale-out 网络。

层级概念图；POD 的实际规模与配置按部署方案变化。NVL72 官方配置为 18 个计算托盘、9 个 NVLink 交换托盘。[R4]

5.1 Rubin GPU：核心微架构与数据通路

336B

晶体管，双 reticle 级计算芯粒，通过 NV-HBI 连接。[R1]

224 SM

配合 896 个 Tensor Cores，面向高密度张量与 Transformer 工作负载。[R1]

288 GB

HBM4 容量；22 TB/s 峰值带宽，约为 Blackwell/Blackwell Ultra 的 2.8 倍量级。[R1]

50 PFLOPS

单 GPU NVFP4 推理峰值（厂商公布口径）；不是任意模型的持续吞吐。[R1]

第三代 Transformer Engine

自动/自适应地在低精度路径中平衡格式、缩放与精度；Rubin 还增加激活稀疏、压缩与 softmax 指令吞吐改进。官方称相对 Blackwell，FP32 指数运算约 2×，BF16/FP16 指数运算约 4×。[R1]

K 维吞吐与 MoE 数据搬运

Rubin 每时钟 K 维处理能力相对 Blackwell 翻倍，减少大矩阵乘内部迭代；TMA 支持内联描述符更新，有利于专家路由后不规则、运行时决定的张量地址与形状。[R1]

细粒度依赖触发

硬件可在 tile 级触发依赖内核/工作，减少必须等待整个生产者内核完成的空档。这与 Megakernel 是互补关系：前者改善混合多内核路径，后者进一步把更多控制移入设备侧。[R1]

设备发起 NVLink counted writes

GPU 可发起带计数语义的远端写入，使接收端通过计数器判定数据到齐；这正适合分块 all-to-all、reduce-scatter 或专家交换中的 producer/consumer 同步。[R1]

5.2 Vera Rubin NVL72：把 72 GPU 变成一个 NVLink 域

层级关键规格对系统设计的含义证据类型

NVL7272 GPU + 36 CPU；20.7 TB HBM4；约 1,580 TB/s 总 HBM 带宽；260 TB/s 机架 NVLink；54 TB CPU 内存可把大模型、MoE 专家和 KV 分布在统一高带宽域；collective 算法与拓扑映射成为一级设计变量初步规格

5.3 变体与异构路径：Rubin、Rubin CPX、Groq 3 LPX

Rubin GPU：通用主引擎

覆盖训练、prefill、decode、MoE、科学计算与通用 CUDA 生态。HBM4 容量/带宽与 NVLink 6 是主系统的共同底座。

Rubin CPX：长上下文专用

官方披露 30 PFLOPS NVFP4、128 GB GDDR7，并宣称相对 GB300 NVL72 的 attention 加速最高 3×。NVL144 CPX 组合 144 CPX GPU、144 Rubin GPU 和 36 Vera CPU，面向百万 token 上下文。[R8]

Groq 3 LPX：低延迟顺序路径

每机架 256 个 LPU、128 GB SRAM、约 40 PB/s SRAM 带宽与 640 TB/s scale-up，定位低延迟推理。官方异构路径让 LPX 处理 decode 中延迟敏感的 FFN/MoE 专家执行，Rubin GPU 继续负责 prefill 与 decode attention；NVIDIA 宣称组合方案吞吐每瓦最高 35×。这是厂商场景化主张。[R7]

不要把 CPX 与 LPX 混为一谈： CPX 更强调长上下文 attention/prefill 类计算；LPX 以片上 SRAM 和确定性数据流追求低延迟顺序推理。实际系统可能让 Rubin 承担通用和大规模并行部分，再把特定阶段卸载到专用加速器。

范围说明：本文聚焦 2026 年 Rubin / Vera Rubin NVL72。官方路线图中的 2027 Rubin Ultra NVL576 属于下一代扩展，不与本报告的当前规格和性能数字混用。[R4]

Megakernel × Rubin

联合分析：Rubin 提供资源，Megakernel 消除编排损耗

以下为基于公开架构的推断，尚需 Rubin 实机验证。

架构推断无公开联合实测
证据边界： 截至 2026-07-23，未发现 Hazy Megakernel、MPK、Event Tensor 或 Ada-MK 在 Rubin GPU/NVL72 上的公开直接基准。下面的“高/中/低”是工程优先级判断，而不是承诺的加速倍数。

7.1 一个更实用的时延模型

T_step ≈ max(T_compute, T_memory, T_comm) + T_residual_sync + T_launch + T_scheduler + T_tail

在可充分重叠的理想情况下，计算、访存和通信由三者最大值决定；真正拉开系统差距的往往是无法重叠的残差。Rubin 主要降低前三项的硬件下界，Megakernel 主要压缩后四项并提高三大资源的重叠程度。若 Rubin 让 T_memory 从毫秒降到百微秒附近，而 T_launch + T_sync 仍维持原尺度，后者占比会变大。

交互式下界估算器

只估算“顺序读取模型字节”的 HBM 时间与“多内核发射间隙”，不包含算力、KV、缓存命中、通信或服务调度；用于理解量级，不是性能预测。

默认 2.48 GB 来自 Hazy Llama-1B 分析中的权重量级；H100 3.35 TB/s 与 Rubin 22 TB/s 分别来自相关公开资料。该计算只展示“硬件越快，固定软件开销越显眼”的机制。[M1][R1]

09
Deployment Playbook

落地方法：从剖析到 Rubin 就绪

9.1 决策树

Q1：GPU 时间线中，launch gap + 内核边界等待是否超过约 10–15%？
否 → 先优化算子、量化、KV 或并行策略；是 → 进入 Q2。
Q2：模型形状是否稳定，且主要目标是 batch 1/低延迟？
是 → 静态 per-SM 指令流或手工 Megakernel 是最快验证路径；否 → 进入 Q3。
Q3：动态性来自连续批处理、MoE 路由还是长上下文？
需要动态全局队列、事件张量/编译器表示与混合内核边界；优先评估 MPK/ETC 类思路，而不是硬编码完整 schedule。
Q4：TP/MoE 中通信是否超过 20–30%？
是 → 把设备侧通信、分块 collective 与计算交错作为一级功能；仅减少发射不够。Rubin 上应重点验证 counted writes 与 NVLink 6 拓扑。
Q5：是否有多租户、抢占或快速模型切换要求？
是 → 采用可退出/可让步的 persistent loop、配额队列与混合执行；不要让一个 Megakernel 无界占住全部 SM。

9.2 五阶段实施

基线固定模型、精度、prompt 分布、batch 策略与 SLA；记录 p50/p95/p99。
剖析Nsight Systems 看空隙和通信；Nsight Compute 看 HBM、occupancy、stall。
最小原型先覆盖 norm/activation/KV 与短 GEMM 周边，不急于吞并所有大算子。
系统集成接入 continuous batching、prefix cache、采样、容错和指标。
Rubin 迁移重新调 tile、SMEM、warp 角色、TMA、NVLink；绝不沿用 H100 常数。

9.3 必须同时观测的 KPI

服务体验

TTFT、TPOT、p50/p95/p99、超时率、抢占恢复、抖动。

吞吐与利用

tokens/s、请求/s、SM active、Tensor Core active、achieved HBM BW、L2 hit。

调度/通信

launch gap、event wait、queue contention、tail wave、comm overlap、collective p99。

经济性

tokens/MW、成本/百万 token、编译/预热时间、显存碎片、运维失败率。

Risks & Boundaries

风险、失败模式与研究空白

资源超配

代码、寄存器或 SMEM 膨胀降低 occupancy；单个慢路径拖住整个驻留网格。需要按 phase 预算资源并允许退回独立内核。

同步正确性

页复用、远端写、事件计数和内存栅栏组合易出现 ABA、越界复用、罕见竞态或死锁。必须有形式化不变量与压力测试。

动态负载偏斜

MoE 热点专家、可变序列和 speculative decode 接受率会使静态 schedule 失衡；动态队列又可能争用。

可移植性

Hopper/Blackwell 的 tile、SMEM、TMA 与指令假设不能原封不动迁到 Rubin。编译器化能减少手工成本，但后端仍需架构特化。

调试与可观测性

一个超大内核在传统 profiler 中像“黑盒”；需要任务级时间戳、事件等待、队列深度、page ownership 和跨 GPU tracing。

多租户与抢占

长期驻留内核可能阻碍其他工作、模型热切换与故障恢复。生产系统必须提供 yield、配额、公平性和看门狗。

软件栈耦合

只替换 kernel 可能破坏 prefix cache、chunked prefill、LoRA、采样、结构化输出和容错语义；端到端集成成本常大于内核开发。

低精度与精度验证

Rubin NVFP4 峰值吸引力很强，但量化策略、缩放、异常值与模型质量必须独立验证；性能不能以质量退化换取。

公开证据缺口

缺少 Rubin 上可复现的 Megakernel、跨 72 GPU 的长期稳定性、独立第三方成本/功耗、以及多租户 p99 数据。

最关键的下一步研究： 在 Rubin NVL72 上，用同一模型、同一服务调度器，比较 CUDA Graph、融合内核、编译式 Megakernel 与混合路径，并同时发布时间线、HBM 利用、collective 占比、p99、tokens/MW 和编译/预热成本。

Glossary

术语表

CTA / Thread Block

在一个 SM 上协同执行的一组线程；Megakernel 常让每个 CTA 长期循环领取任务。

Warp Specialization

让不同 warp 固定承担加载、计算、写回、通信或控制，以构造软件流水。

TMA

Tensor Memory Accelerator，用描述符驱动多维张量搬运；Rubin 的内联描述符更新更适合运行时动态布局。

HBM4

Rubin GPU 使用的高带宽显存；容量和带宽影响权重、KV 与中间激活的数据下界。

NVLink-C2C

Vera CPU 与 Rubin GPU 之间的高带宽一致性互联，官方规格为 1.8 TB/s。

Counted Write

远端写入与到达计数结合；消费者可在计数满足后继续，适合设备侧细粒度 producer/consumer。

TP / EP

Tensor Parallel / Expert Parallel；前者切分张量计算，后者切分 MoE 专家，均依赖高效 collective。

TTFT / TPOT

首 token 时间 / 每输出 token 时间。长上下文通常关心 TTFT，实时交互尤其关心 p99 TPOT。

Wave Quantization

任务数量不是并行资源整数倍时，最后一波只有部分 SM 工作，产生尾部浪费。

Roofline

用计算峰值与内存带宽共同约束性能的上界模型；Megakernel 还需额外考虑发射、同步、调度与通信残差。
