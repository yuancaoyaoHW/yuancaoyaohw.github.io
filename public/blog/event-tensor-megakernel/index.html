Megakernel 与 Rubin：软件抽象与硬件底座

Megakernel 是 GPU 编程模型——把成百上千个小 kernel 融合成一个 persistent kernel； Rubin 是 NVIDIA 下一代 GPU 架构——面向"推理工厂"的整机平台。 两者不构成"区别"，而是软件与硬件两个层级的互补关系。本文整理两者的核心机制， 并深读 megakernel 侧的权威原论文 Event Tensor（MLSys 2026）。

2026-07-22 arXiv + 知乎 Event Tensor
核心定性 · Positioning

不是"区别"，是两个层级

Megakernel 回答的是 "kernel 怎么组织"——把多个算子融合进一个 persistent kernel，消除 launch overhead 和 kernel 边界同步。Rubin 回答的是 "kernel 跑在什么上"——下一代 GPU 芯片的物理规格与整机平台。一个是图灵机上的算法，一个是物理芯片。

真正有意义的问题是：它们各自解决什么瓶颈，以及 Rubin 的硬件特性如何为 megakernel 这种软件思路提供更好的底座。

软件层 · Megakernel

把成百上千个小 kernel 融成一个

解决的瓶颈：LLM 推理的"微核病"

LLM 前向被拆成太多太小的 CUDA kernel（RMSNorm 一个、MLP 一层一个…），每个 kernel 都要：

  • launch overhead——CPU 提交→GPU 启动，每次几 μs，成千上万次累加成主开销
  • kernel 边界强制同步——GPU 排干、结果写回 HBM、下一个 kernel 再读，带宽被中间结果反复读写吃掉
  • 粗粒度流水线——kernel 之间只能串行，SM 在等的时候空转

极端低延迟场景下，"开会"（launch）的时间反而比"干活"长。

核心思路

把整个 Transformer Layer 甚至整个前向融合进一个持续运行的 persistent kernel，在 GPU 内部用协作组网络（grid.sync() / cooperative groups）做细粒度流水线，数据尽量留在寄存器/shared memory，不到万不得已不落 HBM。

关键挑战（为什么难做）

  1. 依赖管理——megakernel 内部不能用 DeviceSynchronize()，得显式建任务图（Task→Event→Task）
  2. 动态依赖——真实 LLM 有动态 batch、MoE routing，手写依赖太复杂
  3. 资源竞争——Attention 要大 SRAM，MLP 要寄存器，挤在一起容易冲突

硬件层 · Rubin

面向"推理工厂"的下一代整机平台

是什么

NVIDIA 下一代 GPU 架构，代号 R200，接 Blackwell。2026 CES / GTC 发布。Rubin 不是一颗芯片，是一整套平台——GPU + CPU + 网络 + 机柜。

平台组成（"七芯片"）

  • Rubin GPU（R200）：核心计算，两颗 reticle 拼接
  • Vera CPU：NVIDIA 自研 ARM 微架构 Olympus（88 核，1.5TB LPDDR5X，每核双超线程空间多线程）——上一代 Grace 用的是公版 Neoverse
  • Spectrum-6 交换机 + ConnectX-9 SuperNIC：Spectrum-X 以太网，专为 AI 工厂的突发/非对称/相关数据流设计

整机形态：Vera Rubin NVL72（机柜级）、HGX Rubin NVL8（节点级）、还有面向太空的 Space-1 Vera Rubin 模组。

解决的瓶颈 / 优化方向

Rubin 不单纯堆算力，而是瞄准三类新负载：

  • Agent AI——多步骤工具调用、长链路推理（需要持续低延迟 token 生成）
  • 万亿参数 MoE——专家分布到多 GPU 后，数据移动和描述符维护成本激增
  • 长上下文 / 逐 token 生成——内存带宽比容量更关键

结构化摘要

背景/目标LLM 推理中 kernel launch overhead（每次 5–10 μs，最快 kernel 仅 2 μs）与 kernel 边界隐式同步（排干→HBM→再读）已成主导瓶颈。megakernel 能消除这两类开销但现有框架不支持动态 batch 与 MoE 路由。本文旨在为动态 megakernel 提供统一编译器抽象。
方法提出 Event Tensor——把"SM 级任务集完成事件"组织成多维张量，作为编译器 IR 的一等对象。基于此构建 ETC 编译器，提供 static 与 dynamic 两套调度变换，自动生成 persistent megakernel。实现于 Apache TVM 之上。
结果8×B200 上：融合 GEMM+RS/AG 最高 1.40× 优于 cuBLAS+NCCL；MoE 层最高 1.23× 优于最优基线；端到端 batch=1 时 1.48× 优于 vLLM；warmup 从 123s/583s 降至 35s（3.5×）。
结论Event Tensor 首次让 megakernel 支持 shape dynamism 与 data-dependent dynamism，实现 AOT 编译、零运行时编译开销。抽象位于 compiler IR 层，不绑定 GPU 代际。

1. 引言

1.1 研究背景与核心问题

GPU 算力持续增长，但传统调度模型里两类开销反而成主导：kernel launch overhead（每次 5–10 μs，最快 kernel 仅 2 μs）和 kernel 边界隐式同步（GPU 排干→HBM→下一个 kernel 再读）。CUDA Graph 减少 launch 但保留 kernel 边界且依赖静态输入；近期 megakernel 工作（Stanford "Look ma, no bubbles!"、CMU Mirage MPK）只支持单 batch dense 推理。本文回答：如何为 megakernel 提供一个既能表达动态形状、又能表达数据相关依赖、还能无缝切换调度策略的统一编译器抽象？

1.2 文献综述与研究缺口

深度学习编译器（MLIR、XLA、TVM）做图级优化但 kernel-by-kernel 执行；vertical/horizontal fusion 与 Rammer/Roller 缺乏细粒度依赖抽象；最新 megakernel 工作只支持单 batch dense、单一调度策略；CuSync 做跨 stream 同步但停留在分离 kernel；Graphene 在单 kernel 内用 thread tensor 建模同步但非多算子融合。缺口：缺乏在编译器 IR 层统一表达细粒度跨算子、且同时处理 shape 与 data-dependent dynamism 的抽象。

1.3 核心命题

隐含假设
把同步事件提升为编译器 IR 中的一等张量（Event Tensor），借助现有编译器对符号形状与张量索引表达式的支持，就能用一个紧致抽象同时表达 SM 级细粒度依赖、动态形状依赖、数据相关依赖，并通过 static/dynamic 两套调度变换自动生成高性能 persistent megakernel，无需运行时重编译或 graph recapture。

2. 研究设计与方法

2.1 范式

构建式系统研究——提出新抽象、实现编译器、在真实硬件上以强基线对比评估。选择该范式因为要验证"抽象能否让编译器自动生成此前需手工特化的 megakernel"。

2.2 数据来源

8×NVIDIA B200（NVLink）、PyTorch 2.8.0、CUDA 13.0。选 B200 因代表 SOTA 硬件，而抽象运作于 compiler IR 层不绑定代际。评估覆盖融合通信+计算、MoE、端到端低 batch serving、warmup 四类工作负载。基线含 cuBLAS+NCCL、TP-Async、Triton-Dist、cuBLASMp、Triton 3.4.0、FlashInfer、vLLM v0.11.0rc2、SGLang v0.5.3rc0。

2.3 三大语言构造

  • Device Function：任务网格，多维坐标参数化，每个 SM 跑一个 tile
  • Event Tensor：多维事件数组，元素含 wait_count，支持 notify()/wait()/trigger
  • Graph Function：含 data tensors + Event Tensors 的计算图，标注输入/输出依赖与坐标映射

2.4 两类动态性

Shape Dynamism：Event Tensor 与 tiled tensor 含符号维度（如 batch B），符号形状图当模板，运行时实例化，免重编译。Data-Dependent Dynamism：通过 runtime index 表达式映射任务坐标→事件坐标。MoE 例子：topk（token→expert 映射）+ exp_indptr（CSR 风格前缀和）动态决定哪些 grouping tiles 通知哪些 expert events、以及每个 expert 触发多少 GroupGEMM tiles。

2.5 两套调度变换

Static Scheduling：宿主预算 per-SM 队列 → persistent main loop → counter-based semaphore + event-triggered wait。对动态 shape 采样代表性形状、未见过复用下一个更大的采样队列；对 data-dependent 取最坏情况重写 notify/wait。Dynamic Scheduling：GPU 上 on-device scheduler 的 push-pop，事件触发后原子 push 消费者任务、空闲 SM 原子 pop 执行。Appendix E 的 early push 策略——producer 一 dispatch 就 push consumer，调度开销与 producer 执行重叠。

Lowering 到最小运行时
Event Tensor → integer tensor（复用现有 tensor 结构），notify() = atomic decrement，wait() = spin-wait counter==0。运行时仅含 integer tensors + scheduler task queue，比传统需要物化整个 task graph 的执行器更轻量。

3. 结果与发现

3.1 融合通信+计算

GEMM+Reduce-Scatter（dynamic scheduler）与 All-Gather+GEMM（static scheduler）均最高 1.40× 优于 cuBLAS+NCCL。动态调度用于通信延迟不可预测的 GEMM+RS，静态调度用于 ring 算法可预测的 All-Gather+GEMM。

3.2 MoE 数据相关负载

Qwen3-30B-A3B（128 experts top-8）单 B200 上，ETC 相对最优基线最高 1.23× 加速（1024 tokens）。机制：data-dependent Event Tensor 打破基线里 GroupGEMM 两阶段之间的全局同步屏障，形成细粒度流水线；on-chip dynamic scheduler 对不规则 token 路由提供负载均衡。

3.3 端到端低 batch serving

Qwen3-30B-A3B TP=1 batch=1 时 1.48× 优于 vLLM、1.20× 优于 SGLang。Qwen3-32B TP=1 batch=1 时 1.15× 优于 vLLM。TP=4 场景 ETC 与 vLLM 持平（0.99×–1.06×）、落后 SGLang——论文归因于工程因素（SGLang CPU 调度器更优、compiler GEMM tiles 不如 cuBLAS 精调）而非抽象根本局限。

3.4 AOT 编译消除 warmup

方法Warmup# JIT Graph Capture

ETC 离线编译单一 shape-generic megakernel（Qwen3-32B 耗时 107s），运行时直接加载。3.5× warmup 加速源于 Event Tensor first-class shape dynamism 赋能 AOT。

3.5 调度权衡

对照"非融合 megakernel"基线（同算子码、仅全局屏障，测纯 inter-kernel parallelism 收益）：MoE 上 dynamic 优于 static（1024 tokens：dynamic 1.08× vs unfused、static 1.04×）。规则 dense transformer（TP=4）上 static 优于 dynamic（1.06×–1.09× vs unfused），dynamic 反而 0.82×–0.89×。结论：数据相关→dynamic，规则 dense→static，两种都需支持。

4. 讨论

4.1 理论贡献

把同步原语（semaphore）从运行时实现细节提升为编译器 IR 的一等张量对象。三重后果：(1) 复用编译器已有的符号形状与张量索引表达式基础设施；(2) static 与 dynamic 调度成为同一 IR 上的可互换变换；(3) 把"producer→event→consumer"任务图范式压缩成张量形式，用单个张量表示成千上万事件。

4.2 实践启示

对推理引擎开发者：提供从"kernel-by-kernel + CUDA Graph"升级到"单一 persistent megakernel + AOT 编译"的路径，利好低 batch 延迟敏感场景。对编译器社区：抽象 DSL-agnostic，可集成 Triton、CuteDSL。对硬件设计者：Event Tensor 的 MoE 数据相关触发机制（topk+exp_indptr）与 Rubin 增强 TMA 在目标上高度对齐。

4.3 局限性

  • Dynamic scheduler 集中式队列：全局内存共享，规模大时可能 contention
  • TP=4 场景不总最优：SGLang CPU 调度器更精调，compiler GEMM tiles 不如 cuBLAS 某些配置
  • 手动标注负担：当前需用户用 TVM-based DSL 写 device function 并手动标注 Event Tensor
  • 代际覆盖：仅 B200 一代验证，未跨 Hopper/Rubin

核心四要素

要素内容
根本问题megakernel 能消除 LLM 推理两大延迟主因，但真实负载的动态性（动态 batch shape + MoE data-dependent 路由）让"手写 megakernel"彻底失效：要么为每个 shape 重编译，要么手写运行时动态依赖（不可维护）。现有编译器对"如何在一个 kernel 内表达并调度动态细粒度依赖"没有抽象。
切入视角别人看到的是"同步是运行时实现细节"，作者看到的是"同步事件本身就是一个张量"。megakernel 把算子拆成海量 tile 级任务后，同步事件天然形成多维结构——把它显式提升为编译器 IR 里的一等张量，就能复用编译器已有的符号形状与索引表达式基础设施。
关键方法两层机制。其一，Event Tensor 抽象：事件=SM 级任务集完成，多维事件数组，wait_count+notify/wait/trigger；shape dynamism 用符号模板+运行时实例化，data-dependent 用 runtime index 表达式（topk+exp_indptr）。其二，ETC 编译器：static/dynamic 两套可互换调度变换，降级为 persistent megakernel；Event Tensor→integer tensor，notify()=atomic decrement。
核心发现用"同算子码、仅把细粒度依赖换成全局屏障"的非融合 megakernel 做对照，证明加速纯粹来自 inter-kernel parallelism。B200 上：融合 GEMM+RS/AG 最高 1.40×；MoE 最高 1.23×；端到端 batch=1 时 1.48× 优于 vLLM；warmup 35s vs 123s/583s（3.5×）；数据相关→dynamic 更优、规则 dense→static 更优。

方法公式化

# 动态 megakernel 的核心公式 动态 megakernel = (细粒度任务图 + Event Tensor) × (static | dynamic 调度变换) # 其中: Event Tensor = 符号形状模板 ⊕ runtime index 表达式(topk + exp_indptr) notify() = atomic decrement wait() = spin-wait(counter==0) AOT 编译 = Event Tensor 的 first-class shape dynamism → 运行时零 JIT、零 graph recapture
一句话总结（核心价值）

本文把 GPU SM 级同步事件提升为编译器 IR 中的一等张量，使 megakernel 首次能自动支持真实 LLM 的动态形状与 MoE 数据相关路由，在不重编译、不 recapture 的前提下生成单一 persistent kernel，相对 vLLM/SGLang 等工业级基线实现端到端 1.15–1.48× 加速与 3.5× warmup 缩减。

一句话总结（大白话版）

以前让 GPU 跑完一步就停一次"开会"再跑下一步，太慢；把所有步骤塞进一个"大 kernel"能快，但只要 batch 一变或碰到 MoE 这种"边跑边才知道下一步找谁"的活儿，手写大 kernel 就抓瞎——这篇论文给编译器装了一个"把任务完成信号排成表格"的通用工具，让编译器自动把动态活儿也塞进一个大 kernel 里跑完。
