    进阶：集群启动控制¶

    概述

    • 持久化核函数（persistent kernel）保持一组固定的 CTA（协作线程数组）或 CTA 集群（cluster）常驻（通常规模设定为每个 SM（流式多处理器）大致有一个活跃的工作所有者，但不依赖保证的 1:1 映射），并让它们循环处理许多输出分块，而非为每个分块启动一个 CTA。
    • 集群启动控制（Cluster Launch Control）是 Blackwell 的硬件机制，允许常驻集群在运行时请求另一个分块。它是一条围绕两条 PTX 指令构建的硬件工作窃取路径：一条指令请求工作，另一条读回请求是否成功。
    • 主要好处是更好的尾部行为。当分块开销不均，或分块数在可用 SM 之间不能均匀划分时，提前完成的 CTA 可以拉取更多工作，而不是闲置等待。

    持久化 GEMM 不把 CUDA 网格（grid）视为固定的"每个输出分块一个 CTA"的启动（launch）。相反，它启动一小组长期存活的 CTA 或 CTA 集群。每一个计算一个分块、前进到另一个分块、再次计算，并持续进行直到输出空间完成。这就是 chap_gemm_advanced 中构建起来的执行模式。

    一旦核函数（kernel）是持久化的，主要的调度（scheduling）问题就变得简单：在一个 CTA 或集群完成其当前分块后，下一个分块从哪里来？

    最简单的答案是一个静态公式。例如，核函数可以从 CTA id 计算分块坐标，然后按网格步长前进。这易于实现，且当所有分块开销大致相同、分块数在 GPU 上均匀分布时效果良好。但调度是在工作实际运行之前决定的。如果少数分块耗时更长，或最后几个分块分配不均，某些 SM 会提前完成其份额而另一些仍在处理尾部。

    集群启动控制（Cluster Launch Control，简称 CLC）改变了该调度模型。它不预先决定整个分配，而是允许持久化集群向硬件网格调度器请求另一个尚未启动的集群的工作。如果请求成功，当前集群接管该集群坐标并计算对应的分块。如果请求失败，则没有更多工作可窃取，循环退出。

    这与线程块集群本身不是同一回事。线程块集群（一起启动的 CTA，具有集群级同步和对分布式共享内存的访问）随 Hopper 引入（chap_background）。CLC 是 Blackwell 的补充，使调度在这些集群坐标上动态化。集群已经是启动的单位；CLC 让一个已在运行的集群取消一个待启动的启动并继承其坐标。

    两条指令¶

    集群启动控制通过两条 PTX 指令暴露。第一条指令向网格调度器发送一个异步请求。第二条指令读取响应。

    请求指令是 clusterlaunchcontrol.try_cancel.async。

    try_cancel 请求调度器取消一个待启动集群的启动，并将该集群的坐标返回给调用者。响应作为一条 16 字节记录写入共享内存。由于请求是异步的，该指令不会等待响应到达。相反，完成通过一个 mbarrier 报告，使用与 TMA 相同的屏障与阶段模型。

    这是一个重要细节，因为它意味着 CLC 不引入新的等待模型。核函数发出请求，将其与一个屏障关联，随后在读取响应前等待该屏障。响应到达通过带字节计数完成的屏障发出信号，风格与其他异步硬件操作相同（见 chap_async_barriers）。

    一旦屏障已触发，核函数使用查询指令。

    第一条查询是 clusterlaunchcontrol.query_cancel.is_canceled。它返回一个谓词，告诉核函数取消是否成功。真谓词意味着调度器找到了一个待启动的集群启动、取消了它并返回了其坐标。假谓词意味着没有剩余工作可取。

    只有当 is_canceled 为真时，核函数才应读取坐标。它用 clusterlaunchcontrol.query_cancel.get_first_ctaid 提取被取消集群的第一个 CTA id。该 CTA id 是一个坐标向量，通常读作 (x, y, z)，核函数将其解码为它接下来应计算的输出分块。

    此协议中没有数值哨兵分块 id。核函数根据谓词分支。如果谓词为真，坐标有效。如果谓词为假，工作窃取循环完成。

    在底层，这一形态直接源于 CLC 所做之事。硬件并非从软件队列分配抽象任务。它是在取消一个尚未发生的集群启动。因此成功的响应包含一个真实的集群坐标。失败的响应仅意味着启动队列已耗尽。

    工作窃取循环¶

    有了这两条指令，持久化调度器就变成一个短循环。

    在循环中的任意时刻，集群负责计算一个分块。在它开始该分块之前，它为下一个分块发送一个 try_cancel 请求。请求异步运行。当调度器处理该请求时，集群计算其当前分块。

    当前分块完成后，集群等待与 try_cancel 响应关联的 mbarrier。响应就绪后，它调用 query_cancel.is_canceled。如果谓词为真，它调用 query_cancel.get_first_ctaid，解码返回的坐标，并将其用作下一个分块。如果谓词为假，则没有更多工作，集群退出。

    在代码形态上，循环为：

    1. 为可能的下一个分块发出 try_cancel；
    2. 在请求在途时计算当前分块；
    3. 等待响应屏障；
    4. 查询取消是否成功；
    5. 要么以返回的坐标继续，要么退出。

    请求的放置正是使循环有用的关键。集群不会等到完成当前分块才请求更多工作。它先请求，再计算。这将调度器请求与有用工作重叠。到当前分块完成时，下一个分块的答案往往已经可用。

    这与持久化核函数在其他地方使用异步拷贝和 Tensor Core 屏障的基本原因相同。核函数避免将长延迟操作直接放在关键路径上。CLC 将同一思想应用于分块调度：尽早请求下一个工作单位，计算当前单位，然后在需要时消费调度结果。

    与持久化 GEMM 的关系¶

    chap_gemm_advanced 中的持久化 GEMM 在主要讲解中使用静态调度器。静态调度器更易解释，因为下一个分块可以直接从循环状态计算。例如，ClusterPersistentScheduler2D 这样的调度器可以在输出分块空间上以网格步长模式分配分块。

    CLC 是该静态分配的动态替代。外层循环保持不变：每个常驻集群反复计算一个输出分块然后前进到另一个。变化的是下一个分块从哪里来。使用静态调度器时，下一个分块由公式计算。使用 CLC 时，下一个分块由硬件工作窃取返回。

    这一差异在启动的尾部附近最为重要。在静态调度中，剩余工作可能分布不均。某些 SM 可能用完了分配的分块，而另一些仍有几个剩余。使用 CLC 时，提前完成的集群请求另一个待启动的集群坐标。只要启动队列中还有工作，提前完成者就会持续拉取更多分块。

    当分块开销不均匀时它也很重要。某些 GEMM 分块可能因边界、掩码、稀疏性、分组调度或围绕主矩阵乘法的融合工作而走不同路径。静态调度假设分块分配在任何这些开销被观测之前就足够好。CLC 不需要该假设。它仅在集群变得可用之后才分配更多工作。

    因此，在 TIRx 中，CLC 可以作为一个动态分块调度器暴露。编程模型不需要改变分块的计算。分块主体与静态调度器所用的持久化 GEMM 主体相同。调度器从"用公式计算我的下一个分块坐标"变为"向硬件请求下一个可用集群坐标"。结果是相同的持久化循环，但以硬件驱动的工作分发取代固定的启动时调度。
